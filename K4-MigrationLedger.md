**`Corpus Role` is not optional.** Without it the system is structurally unable to clean a corpus: a corpus-prep task's face outputs contradict the project content **by construction**, which under `SPECIFICATION` semantics is Structural Shear, which raises, which exhausts the cap, which halts the run **before** the cleaned corpus is ever written at C9. The mess triggers the safety limit that aborts the cleaning.

This is a **use/mention** distinction, and it closes the whole class — corpus prep, refactors, migrations, spec rewrites: anything that operates on its own inputs declares `MATERIAL`. One tag, no exception handler, no second special case waiting to be discovered.

**Halt condition, corrected:** not *"files declared, no routing table."* It is **"files declared, not located."** Same shape as the AbsentVar — not a missing lookup, a missing coordinate.

## 5. VALIDATOR SPLIT — proposed

`K4-AlgebraicInputValidation` → **`K4-AlgebraicTaskValidation`** + **`K4-AlgebraicTaskDocumentValidation`**

The tell is already in the current doc: Phase 1's **Conflict Rule** is one line —

> *"Where the prompt asserts a pole the corpus contradicts, this is an audit `flagged` condition."*

— holding an entire second discipline in a footnote.

| | Task Validation | Document Validation |
|---|---|---|
| **Object** | one prompt | N documents |
| **Converges on** | a coordinate | a **location** (per-file pole coordinate) + distillation state |
| **Failure modes** | missing pole, unconfirmed inference | **misrouting**, **prohibition breach** |
| **Lifecycle** | per task | corpus revised without the task changing |

Both inherit: binding gate, state header, turn envelope. **Only Document Validation emits the Payload.**

**Two checks that exist only once the corpus is in scope:**

1. **Misrouting is structurally detectable.** Content filed under $P$ that reads as material constraint *is* an $R$-fact in the wrong drawer. Consequence: the Earth face is measured on constraints it was never shown, and hallucinates them to satisfy its equations. Same cascade, entered through the corpus instead of the prompt.
2. **The corpus must not resolve the AbsentVar.** If the operator signed off carrying $I$ absent, and the corpus contains a fully-specified integration schema — the corpus contradicts the Payload's own Strict Prohibition. The Swarm is forbidden to resolve $I$ and simultaneously handed $I$. Impossible to catch in a prompt-validator; trivial against a locked coordinate.

**Regimes (operator's choice; we comment, we do not dictate):**

| Regime | Shape | Failure mode |
|---|---|---|
| **Sharded** | one file per pole (+ common) | misrouting |
| **Outline** | one doc, pole-tagged sections | soft boundaries, cross-bleed |
| **Monolithic** | everyone parses everything | no routing signal; every face re-derives relevance every cycle; operator burnout — the thing this was built to prevent |

## 6. OPEN SOCKETS

### 6.1 — Hamiltonian Path Selection *(load-bearing)*

**Settled:** the stance is locked upstream → the stance fixes the AbsentVar → selection ranges over the **active variables only**, three not four. The stance equation names its own arguments and output, which constrains order further.

**Open:** among the stance equation's arguments, **which first**.

**Criterion:** *least-developed* and/or *most-urgent*. These are two different signals —

* **least-developed** = a **deficit** signal (where specification is thinnest)
* **most-urgent** = a **dependency** signal (what everything else is blocked on)

They coincide often and **diverge sharply**. The thinnest pole may be one nothing needs yet. The blocking pole may be well-specified — and blocking *because* it is specified enough that others cannot proceed without consuming it.

Reduces to a choice over **two**. *(Elsewhere: the law of intuition.)*

**Current placeholder:** `P → U → I → R` — in use because the faces have nothing to show yet. A stud cut to fit the socket, not an answer.

### 6.1b — Stance Selection *(across cycles)* — **NEW**

The Braid rule yields the **next active pair**. A pair does not name a stance.

Reading the 12-stance table: **every active pair admits exactly two stances** (6 pairs × 2 = 12), differing only in which of the two *non-traversed* poles is **home** (the metric) and which is **absent** (held).

> *Of the two poles you are not traversing — which do you measure by, and which do you hold?*

**Same shape as 6.1.** Path selection reduces to a choice over two; stance selection reduces to a choice over two. Whether that is **one mechanism with two applications** or a coincidence of the table is **not settled** — and should not be built on until it is.

**Who chooses is also open.** The Controller runs one cycle and yields. The PTR carries the pair and the two candidates. Selection currently falls to the **caller** (Validation). Whether it should is a design question, not a wiring one.

### 6.2 — The Bridge *(interactive; scale-aware)*

Not a Controller phase. It is **interactive**, and the Controller has zero operator gates.

* Performs **fractal descent through Powers-of-10 scales**.
* Navigates toward the termination condition **the operator walked in already holding** — implicitly specified, not stated.
* Rationale: a sane operator does not go for broke. They level up from something small.
* **Pipeline position: `Task → Documents → Bridge → Payload`.** The Bridge is a **tail**, not a middle joint. *(Reordering withdrawn: project docs are project docs. Rendering/modelling resolution is orthogonal to the corpus. A doc does not come in and out of scope as you descend.)*
* **Lemma docs pending.**

### 6.3 — Concurrency

The Controller compiles a well-defined driver for face-logic and runs it once. It knows nothing of concurrent instances, result selection, or best-match.

Later: *"run 5 instances of this face, select best."* A change to the **executor**, not to the Controller. `last-one-wins` → `selected-one-wins`. Same slot, same stamp, same readers. **The notch is cut; nothing needs to move to fill it.**
